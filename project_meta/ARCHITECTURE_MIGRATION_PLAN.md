# План архитектурной миграции (Architecture Migration Plan)

Данный документ представляет собой пошаговый план реализации архитектурных улучшений, описанных в `FUTURE_ARCHITECTURE.md`.

## 1. Core Engine (Безопасность пайплайна и Fail Fast)
Текущая реализация полагается на `sync.WaitGroup`, не прерывая соседние потоки при фатальных ошибках кодирования одного сегмента.
- **Задача:** Рефакторинг `internal/engine/engine.go`.
- **Действие:** Обернуть фазы `Render` и `Encode` функции `Run` в `golang.org/x/sync/errgroup`.
- **Ожидаемый результат:** Если энкодер или рендерер возвращает ошибку, контекст всей группы `errgroup` немедленно отменяется (`cancel`), предотвращая пустую трату ресурсов на оставшиеся фреймы.

## 2. Оптимизация системы управления памятью (Memory Optimization)
Бэкпрешеринг через `sync.Cond` вызывает накладные расходы на "пробуждение" (broadcast/thundering herd problem).
- **Задача:** Модификация `internal/system/memory.go`.
- **Действие:** Замена `sync.Mutex` и `sync.Cond` на взвешенный семафор `golang.org/x/sync/semaphore`.
- **Действие 2:** Использование `semaphore.NewWeighted(int64(maxMemoryBytes))` и методов `Acquire/Release` для плавного приостановления горутин без активного поллинга.
- **Тесты:** Добавление `memory_test.go` для валидации потокобезопасности.

## 3. Масштабируемая конкатенация (Chunked Concatenation)
Метод склейки падает/завешивает FFmpeg при огромном числе сегментов (`filter_complex_script` становится слишком большим).
- **Задача:** Обновление `internal/video/ffmpeg.go`.
- **Действие:** Модификация логики `Concatenate` для нарезки массива `segments` на чанки (например, по 50-100 сегментов).
- **Действие 2:** Обработка каждого чанка во временный файл (`chunk_01.mp4`, `chunk_02.mp4`) со своими переходами.
- **Действие 3:** Финальная склейка чанков через `concat demuxer` (без ресемплирования/декодирования) в итоговый файл.

## 4. Абстрагирование Computer Vision (Analyzer Interface)
Вся логика компьютерного зрения сейчас — это функции на чистом Go, привязанные к процессору.
- **Задача:** Расширение `internal/analyzer`.
- **Действие:** Внедрение контракта `type Analyzer interface { Analyze(img image.Image) ([]Block, error) }`.
- **Действие 2:** Инкапсуляция текущих алгоритмов Собеля и контраста в структуру `GoNativeAnalyzer`.
- **Ожидаемый результат:** Инверсия зависимостей (Dependency Inversion), позволяющая легко подключать воркеры на базе Python (YOLO) или OpenCV через CGO без изменения движка.

## 5. Очистка архитектуры (Builder Pattern & Logging)
Файл `main.go` перегружен флагами и конструированием `Config`.
- **Задача:** Выделение конфигурации в отдельный слой `internal/config/builder.go`.
- **Действие:** Реализация паттерна Builder (`NewBuilder()`, `WithInput()`, `Build()`). Инициализация проекта в `main.go` сводится к минимуму (CLI-адаптер → Builder → Run).
- **Действие 2 (Опционально):** Замена `log.Printf` на структурное логирование (`Go 1.21 log/slog`) для повышения прозрачности и парсинга метрик.

## Метрики успешности (Verification Plan)
1. Остановка рендера: Инкжектирование "битого" кадра мгновенно обрывает всю сборку без генерации остальных 100 слайдов.
2. Потребление RAM: `Activity Monitor` доказывает, что `semaphore` удерживает RAM строго в пределах заданного бюджетного лимита без всплесков CPU (100%->0% locking time).
3. Долгие видео (1 час+): Сборка 1000 слайдов успешно завершается (чанки) без падения FFmpeg по лимиту `filter_complex_script`.
